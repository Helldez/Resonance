# Resonance — Security Posture

This document is the running audit of the cryptographic identity material
that Resonance stores on-device, and the threat model it survives or
doesn't. It is updated whenever an identity / routing key surface is
added or changed.

The product posture (see `AGENTS.md`) is **privacy-as-constraint, no
central server, no telemetry**. The security posture below follows from
that.

---

## What identities exist on a Resonance device

| Identity | Purpose | Key type | Where it lives |
|---|---|---|---|
| **Author identity** | Signs every `SignedRecord` so peers can verify authorship of posts and responses. Public half = `PeerId`. | Ed25519 keypair via `@noble/ed25519` | Private seed (hex) in AsyncStorage under `StorageConfig.identityKvKey`. |
| **Hyperswarm noise identity** | DHT-routable identity used by Hyperswarm/HyperDHT for peer discovery and `joinPeer` sticky connections. | Ed25519 keypair (NaCl-style 64-byte secret) | `swarm-keypair.json` under `StorageConfig.corestoreDir`. Generated by the Bare worker on first launch. |
| **Outbox Hypercore key** | Identifies the per-peer append-only feed of signed records. Auto-derived by Corestore from a deterministic seed. | Ed25519 (Hypercore-internal) | Inside Corestore's on-disk state. |

The **Author Ed25519** and the **Hyperswarm noise Ed25519** are *two
different keypairs*. Compromise of one does not directly compromise the
other, but in the current threat model they share the same storage
medium so leakage tends to be correlated.

## Audit — current findings

### HIGH severity (fixed in this commit)

#### F1 — Android Cloud Backup was exfiltrating secret material

`AndroidManifest.xml` had `android:allowBackup="true"` (Expo default).
Android's Auto Backup syncs the app's sandboxed data dir — including
AsyncStorage and the Corestore directory — to the user's Google Drive
account every ~24 hours. This means the Ed25519 author private key AND
the Hyperswarm noise secret were being uploaded to Google's servers
encrypted under the user's Google account key, not under any local
device key.

**Fix**: `app.json` now sets `"android": { "allowBackup": false }`. The
next `expo prebuild` regenerates AndroidManifest with
`android:allowBackup="false"`. Backups stop. Existing backups should be
purged via Google Drive → Backups → delete app data.

### MEDIUM severity (deferred, tracked)

#### F2 — Author Ed25519 private key stored in AsyncStorage (plaintext)

`Ed25519Identity.loadOrCreate()` persists the hex-encoded private seed
via the `AsyncStorageKv` adapter. AsyncStorage on Android writes
plaintext to the app sandbox. Risks:

- **Rooted device**: any process with root can read it.
- **Forensic image of the device**: same.
- ~~**Cloud backup**: mitigated by F1.~~

**Mitigation status**: F1 closes the realistic remote attack vector. The
remaining risk requires physical access or root, which is outside the
threat model for an MVP that does not yet handle financial value.

**Planned fix**: migrate to `expo-secure-store`, which wraps Android
Keystore. On supported devices (Android 6+ with hardware-backed key
store) the key encryption material lives in the TEE and the private
seed becomes inaccessible even with full sandbox read access.

#### F3 — Hyperswarm noise secret stored in `swarm-keypair.json` (plaintext)

Symmetric to F2. The keypair is generated and persisted by the Bare
worker (`bare/p2p.mjs:loadOrCreateSwarmKeyPair`). Same mitigation
status.

**Planned fix**: have the RN side generate the keypair via
`expo-secure-store` and pass it to the Bare worker at `init` time.
Bare worker stops self-managing.

### LOW severity / informational

#### I1 — Embeddings travel in cleartext

Each `PostBody` includes the 256-dim Float32 embedding of the post.
Peers in the same bucket receive the embedding alongside the text. A
listener in the same bucket can therefore reconstruct the **semantic**
content even without reading the text, and could in principle correlate
embeddings against an external corpus to de-anonymise.

This is explicitly accepted in `AGENTS.md` ("fuzzy PSI / homomorphic
similarity" is out of MVP scope). Documented here for completeness.

#### I2 — DHT announces are publicly observable

Anyone with a Hyperswarm DHT crawler can see "peer P announced topic T"
events. With `MatchingConfig.lshBits = 8` the topic space is 256
buckets, so a global observer can map every peer to one of 256 coarse
semantic regions. This leaks profile granularity, not content.

#### I3 — Bare worker shares the JS process

`react-native-bare-kit` runs the Bare worker in the same OS process as
the React Native JS bridge. A compromise of either side can read the
other's memory. There is no in-process sandbox. This is a property of
the Bare runtime, not a Resonance bug.

#### I4 — Sybil attackers can listen but not impersonate

A bad actor can generate arbitrary Ed25519 identities and Hyperswarm
keypairs and join any bucket. They can read every replicated record in
that bucket (signatures are public). They **cannot** publish records as
another peer because that requires the corresponding private key.

Sybil-resistance for *posting* is provided by Ed25519. Sybil-resistance
for *listening* is not provided and is not desired in a permissionless
network.

## Positive findings

- **Ed25519 + SHA-512** via `@noble/ed25519` and `@noble/hashes/sha512`
  (well-reviewed pure-JS implementations, audited by Cure53 in 2024).
- **All inbound records verified**:
  `AppContainerContext.persistRecord` recomputes the canonical digest
  and verifies the Ed25519 signature against the claimed `author`
  before persisting (`Ed25519Identity.verify`).
- **Hypercore append-only ordering** prevents within-feed replay.
- **No central server, no analytics, no telemetry**. Only outbound
  traffic at runtime is Holepunch DHT (public infrastructure). Model
  downloads from HuggingFace happen once, unauthenticated, public URLs
  pinned in `HttpModelSources.ts`.
- **Minimal Android permissions** (`INTERNET`,
  `ACCESS_NETWORK_STATE`, `POST_NOTIFICATIONS`, `FOREGROUND_SERVICE`).
- **No regex in security-sensitive paths** (per AGENTS.md). Canonical
  serialiser is byte-level deterministic.

## Threat model — what Resonance defends against

| Adversary | Defended? | Mechanism |
|---|---|---|
| Network observer on the same Wi-Fi | Partial | Hyperswarm uses Noise XX (authenticated encryption per connection). The DHT-level metadata (which bucket you announce) is in the clear. |
| Peer in the same semantic bucket | Read-only | Sees every replicated record (post text + embedding). Cannot impersonate other peers (Ed25519). |
| Malicious peer crafting forged records | Yes | Signature verification rejects them on ingest. |
| Lost/stolen unlocked device | No | Plaintext keys in AsyncStorage, plaintext noise keypair on disk. F2/F3 deferred. Mitigation: device lock screen. |
| Cloud backup exfiltration of secrets | **Yes (after F1)** | `allowBackup=false`. |
| Sybil flood of fake peers in a bucket | Partial | They can listen, cannot impersonate. Hyperswarm caps peers per topic (~24-64) so flood doesn't DoS the device. |
| Compromised mobile OS (root malware) | No | Out of scope for a userspace app. |

## Roadmap

| Phase | Item |
|---|---|
| **Done (this commit)** | F1 — disable Android Auto Backup. |
| Phase 2 | F2 — migrate Ed25519 private key to `expo-secure-store`. |
| Phase 2 | F3 — migrate Hyperswarm noise keypair to `expo-secure-store`, passed to Bare worker at init. |
